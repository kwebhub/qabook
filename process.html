Software Development Life Cycle (SDLC) & Software Test Life Cycle (STLC)

Idea

Product Owner (PO) - Владелец продукта озвучивает идею

QC проверяет:

  • Сможем ли мы это протестировать?
  • Зачем это нужно?
  • Существует ли более простая альтернатива?

STLC: Учим английский

Analysis

Business Analyst (BA) or System Analyst (SA) - Бизнес аналитик формирует бизнес требования:

  • Software Requirements Specification (SRS)
  • Software Requirements Document (SRD)
  • Business Requirements Document (BRD)
  • User story

QC: Все ли варианты развития событий учтены?

  • Test Plan
  • Acceptance plan
  • Acceptance criteria

STLC: Этап планирования

  • Определить список действий (активностей), которые нужно сделать, чтобы проверить максимально все возможные варианты работы продукта исходя из имеющихся времени и средств.
  • Каждое действие имеет свою цель, задачи, методы, ограничения.
  • Для каждого действия указывается время выполнения в общем графике тестирования.
Architecture & Design

Activity:

  • Архитектор составляет технические требования на основании бизнес требований.
  • Дизайнер создаёт дизайн.
  • Верстальщик верстает прототип: Layout.
  • SEO специалист разрабатывает план.
  • DevOps обеспечивает среду для разработки, тестирование и релиза.

QC create:

  • Check lists
  • Smoke Check lists
  • Test Data
  • Test Cases
  • Test Suites

STLC: Этап реализации

  • Описать шаги и ожидаемый результат по каждому тесту и сгруппировать их по общим начальным условиям или одинаковым шагам выполнения.
  • Создать и/или подготовить необходимое тестовое обеспечение для выполнения тестов.
Testing

Выполнение тестов и исправление багов:

  • Разработчики исправляют баги
  • Тестировщики проверяют снова и снова
  • DevOps: Release candidate build
  • Acceptance Testing: Владелец продукта тестирует продукт

Testers execute:

  • Check lists
  • Test Cases
  • Defect
  • Reviews
  • Issue reports
  • Bug Reports
  • Acceptance testing

STLC: Этап выполнения

  • Запустить выполнение тестов по расписанию.
  • В чек-листах отметить выполнение шагов и полученный результат.
  • В Дефекте по каждому тесту напротив ожидаемого результата указать полученный и отметить выполнен тест или провален.
Release

DevOps deploy release build

QC create:

  • Test Summary Report
  • Change Requests

STLC: Этап завершения

  • Описать в итоговом отчёте условия и ход выполнения действий из плана тестирования с комментариями и пояснениями.
  • В запросе на улучшение указать что лучше добавить или переделать в продукте с точки зрения удобства пользования.
Support

Feedback from customers:

  • Успокоить пользователя
  • Исправить или передать проблему разработчикам
  • Агрегировать и систематизировать пожелания потребителей

Уровни поддержки:

  • Успокоить пользователя
  • Исправить или передать проблему разработчикам
  • Агрегировать и систематизировать пожелания потребителей

QA create:

  • Test Progress Report
  • Test Summary Report

STLC: Этап мониторинга и контроля

  • Мониторинг тестирования - это непрерывное сравнение фактического хода работы с планом тестирования.
  • Контроль тестирования - это принятие мер, для достижения целей плана тестирования.
  • План тестирования может быть обновлен.

Work process

Онбординг

2 - 4 недели: чек-лист с дедлайнами, сделать презентацию с анализом гайдлайнов, окружения и документации

  • Знакомство с командой.
  • Знакомство с проектом и продуктом.
  • Знакомство с заказчиком
  • Интуитивное тестирование (Ad-hoc testing).
  • Целеполагание.
  • Изучение и анализ требований.
  • Изучение тестовой документации.
  • Получение необходимых доступов и техники.
  • Знакомство с процессами в Bug tracking system (BTS).
  • Презентация и внедрение схемы с участием тестировщика на каждом из этапов разработки
  • Календарь релизов

Уточнить:

  • какая документация используется на проекте?
  • есть ли шаблоны документации не проекте, по которым работает команда?
  • есть ли примеры тестовой документации, а также кейсы для тренировки?
  • где храниться документация?
  • какой формат управления документацией?
  • кто выставляет приоритеты?
  • есть ли разница в составлении документации для фронта и бэка?

Источник: https://www.atlassian.com/blog/inside-atlassian/jira-qa-process

Написать тест план (тестовую стратегию)
  • Выбор тестовой документации.
  • Разделение полномочий.
  • Выбор видов тестирования.
  • Описание основных процессов тестирования (QA Flow).
  • Критерии входа и выхода из тестирования.
  • Выбор отслеживаемых метрик.
  • Согласование.
  • Настройка тестового окружения
Настройка окружения
  • Анализ продукта и текущего CI/CD процесса
  • Выбор багтрекинговой системы
  • Выбор системы по управлению тестовой документацией
  • Участие в выборе и разработке тестовых стендов
  • STAGE env только для тестировщиков
  • Сделать отчет о проведенном анализе
Разработать чек-листы (тест кейсы)
  • Определить подходы по формированию наборов тестов.
  • Определить критерии для регрессионных и дымных тестов.
  • Разработка минимального количества тестовых наборов: smoke, NFT, regression, API, automation.
  • Матрица трассировки.

Источник: https://github.com/reportportal/reportportal/issues/507

Провести тестирование
  • Четкое и ясное понимание QA flow всей командой
  • Процесс тестирования на всех этапах разработки
  • Эстимация задач должна проводиться с учетом времени на тестирование
  • Не прохождение SMOKE – остановка тестирования
  • Тестирование новой функциональности.
  • Ретест.
  • Регрессионное тестирование.
  • Все баг-репорты должны быть в системе, никаких устных договоренностей с разработчиком
  • Новый билд только после фикса всех критичных дефектов
Подготовить отчеты
  • Отчет по результатам тестирования в конце итерации и/или перед релизом
  • Отчет по результатам тестирования должен содержать решение команды тестирования о возможности к выпуску продукта в релиз
  • Отчет по результатам тестирования с использованием метрик.
  • Рекомендации и риски.
  • Оценка общего качества продукта.
  • Количество обнаруженных багов
  • Количество написанных тест-кейсов
  • Диаграммы по PRIORITY обнаруженных багов
  • Отслеживание изменения метрик в динамике
